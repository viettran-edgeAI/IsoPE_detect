# Embedded Phase Design (Feature Extractor + Model Engine)

## 1) Scope and Objective

The embedded phase converts development outputs into a deployable endpoint pipeline with two independently versioned modules:

1. **Feature Extractor** (`embedded_phase/src/feature_extractor/`)
2. **Model Engine** (`embedded_phase/src/model_engine/`)

The runtime objective is deterministic static PE scoring with strict parity to development-phase decisions.

## 3) Feature Extractor Module Design

## 3.1 Responsibilities

- Parse PE files using LIEF.
- Emit **raw selected features** in exactly the same order as `optimized_feature_list.json`.
- Return extraction metadata (`parse_ok`, bounded error text, processing time).

## 3.2 Compile-time feature locking

- `embedded_phase/src/feature_extractor/generate_compiled_features.py` converts `optimized_feature_list.json` into `compiled_feature_config.hpp`.
- `build.sh` uses `OPTIMIZED_FEATURE_LIST_PATH` by default, pointing to:
  - `development_phase/results/optimized_feature_list.json`

## 3.3 Important rule

The feature extractor must **not** perform model scaling or threshold logic. It is a pure feature production stage.

---

## 4) Model Engine Module Design

## 4.1 Responsibilities

- Build/train C++ Isolation Forest from `development_phase/data/optimized/*_optimized.*` and `model_engine_config.json`.
- Apply final threshold policy for decisions.
- Expose score + verdict APIs for endpoint use.

## 4.2 Artifact contract

`model_engine_config.json` is the canonical source for:

- hyperparameters
- thresholding metadata
- scaler parameters
- evaluation metadata
- deployment scaling policy

## 4.3 Scaler strategy (applied)

The chosen policy is:

- Keep scaler parameters in artifact for audit/parity.
- Disable runtime normalization in model engine inference.
- Fold scaling into tree thresholds at build time.

This is recorded in `model_engine_config.json` under `deployment_scaling`.

---

## 5) Build-Time Threshold Folding Technique

For a split on feature `j` trained in standardized space:

- standardized value: `z_j = (x_j - mean_j) / scale_j`
- split test in trained model: `z_j <= t_scaled`

Convert to raw-space threshold once during build:

- `x_j <= t_raw`
- `t_raw = t_scaled * scale_j + mean_j`

### Practical notes

- `scale_j` must be non-zero (guard with epsilon if needed).
- Keep split direction unchanged.
- Fold every tree node threshold using the node feature index.
- After folding, runtime inference uses raw extractor features directly.

### Why this policy

- Removes per-sample normalization overhead in runtime path.
- Preserves decision equivalence to standardized-space model (up to floating-point tolerance).
- Keeps reproducibility because scaler vectors remain stored in artifact.

---

## 6) End-to-End Embedded Pipeline

1. Stage-3 run produces:
   - `optimized_feature_list.json`
   - `model_engine_config.json`
2. Feature extractor build locks selected feature schema at compile-time.
3. Model engine build loads config, trains from optimized datasets, folds thresholds.
4. Runtime path:
   - extractor emits raw selected features
   - model engine scores raw features using folded trees
   - threshold policy emits anomaly decision

---

## 7) Verification and Parity Gates

## 7.1 Functional gates

- Feature count must match (`40` currently).
- Feature order hash from extractor must match `optimized_feature_list.json`.
- No missing/extra features at model-engine input boundary.

## 7.2 Numerical gates

- Compare Python vs C++ scores on fixed sample set.
- Track max/mean absolute score error.
- Track decision agreement at production threshold.

## 7.3 Operational gates

- Enforce deterministic seeds for reproducible training.
- Keep sealed-test policy from development phase (test is holdout only).
